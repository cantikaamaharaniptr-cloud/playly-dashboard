### 🔒 Keamanan (5)

#### [#1] Password hash lemah (SHA-256 + salt global tetap) disimpan dalam blob kv.value dan sempat terbaca lewat anon key publik
- 🚦 **GENTING** · ⏱ ±multi-day · ⚠ Risiko fix bikin rusak: **TINGGI** · 📍 `public/legacy/script.js, supabase/migrations/0013_kv_orphan_leak_fix.sql, app/api/kv/sync/route.ts` (script.js:1148-1151; 0013:1-14; kv/sync:103-112)
- 🔍 **Masalah:** Model auth legacy menyimpan password sebagai hash di field value.password pada baris kv (key playly-account-{email}). Hashing dilakukan di sisi klien: skema h2 = SHA-256('playly-pwd-salt-v1' + plaintext) memakai SATU salt global tetap yang hardcoded di JS publik (ADMIN_PASSWORD_HASH dan DEFAULT_RESET_PASSWORD_HASH juga hardcoded). Migration 0013 mendokumentasikan kebocoran NYATA: baris playly-account-{email} dengan user_id=NULL dapat dibaca oleh siapa pun yang memegang anon key (anon key bersifat publik, ada di browser), mengekspos field 'password' (hash). Contoh ditemukan 27 Jun 2026: playly-account-cantikaputri@gmail.com dan playly-account-admin.playly2@gmail.com.
- 🎯 **Kenapa penting:** SHA-256 tanpa salt per-pengguna dan tanpa key-stretching sangat rentan rainbow-table/brute-force offline; salt global tetap berarti dua user dengan password sama menghasilkan hash identik dan dapat di-precompute. Karena hash bisa terbaca lewat anon key publik, penyerang dapat mengunduh massal hash lalu crack offline (OWASP A02 Cryptographic Failures). Hardcoded admin hash + salt di JS publik berarti penyerang dapat memverifikasi tebakan password admin secara lokal.
- 💥 **Dampak bila dibiarkan:** Kompromi kredensial massal: pengambilan hash via anon key + cracking offline → pengambilalihan akun (termasuk akun admin). Kebocoran sudah terjadi di produksi (didokumentasikan migration 0013), sehingga ini bukan hipotetis.
- 🛠 **Cara perbaiki:**
    1. Hentikan penyimpanan password/hash di tabel kv sepenuhnya; jadikan Supabase Auth (auth.users) satu-satunya sumber kredensial (sudah ada jalur bridge B6b/P2-P3 untuk drop SHA-256).
    2. Pastikan migration 0013 Step 1 (stamp user_id) dan Step 2 (perketat policy anon) benar-benar sudah di-apply di produksi; verifikasi 0 baris hasil query 'select key from public.kv where user_id is null and key like playly-account-%@%'.
    3. Lakukan rotasi/force-reset password untuk akun yang hash-nya mungkin sudah ter-ekspos, terutama akun admin yang hash-nya hardcoded.
    4. Hapus konstanta ADMIN_PASSWORD_HASH/DEFAULT_RESET_PASSWORD_HASH dan skema h2 salt-global dari kode klien.
    5. Tambahkan filter agar field sensitif (password) tidak pernah ikut tersinkron ke cloud melalui kv/sync.
- ✅ **Hasil cek-ulang:** NYATA sebagian, dengan koreksi penting pada karakterisasi kripto. TERKONFIRMASI: (1) Hash admin/reset hardcoded di JS publik adalah h2 = SHA-256('playly-pwd-salt-v1'+plaintext) dengan salt global tetap — script.js:1148-1151 + hashPasswordV2 di script.js:23366-23370, salt 'playly-pwd-salt-v1' di 23335. (2) Field value.password (hash) memang ikut ter-sync ke baris kv playly-account-{email} (cloud-sync.js:360-378 mengakui 'password hash' di value). (3) Kebocoran anon-readable NYATA dan terdokumentasi konkret di migration 0013_kv_orphan_leak_fix.sql:1-14 dengan contoh akun (cantikaputri@, admin.playly2@) per 27 Jun 2026 — orphan user_id=NULL lolos kv_anon_platform_read (0012:98-101). KOREKSI: klaim 'password disimpan sebagai SHA-256 + salt global' TIDAK akurat untuk akun normal. hashPassword() runtime memakai h3 = PBKDF2-SHA256, 100000 iterasi, salt random 16-byte per akun (script.js:23348-23362); h2 hanya legacy-verify dan login berhasil auto-upgrade ke h3 (23846-23847). Jadi hanya hash admin/reset hardcoded + sisa akun h2 lama yang benar-benar SHA-256+salt-tetap. Severity critical tetap tepat karena leak hash password lewat anon key publik terkonfirmasi nyata (OWASP A02/A01). Risk high (bukan certain) karena 0013 menyatakan code-fix sudah live (cloud-sync v559 tak fallback anon) + data orphan di-stamp; eksposur masa-lalu nyata, eksposur saat-ini bergantung apakah 0012/0013 sudah benar-benar di-apply ke DB (tak bisa diverifikasi dari kode, dilarang sentuh DB).

#### [#2] Tidak ada rate limiting di endpoint autentikasi sensitif (auth/bridge signin+signup, auth/verify)
- 🚦 **GENTING** · ⏱ ±2hr · ⚠ Risiko fix bikin rusak: **SEDANG** · 📍 `app/api/auth/bridge/route.ts, app/api/auth/verify/route.ts` (bridge:28-88; verify:31-85)
- 🔍 **Masalah:** Tidak ditemukan satu pun mekanisme rate limiting di seluruh app/api (grep rate/limit/throttle/upstash → nihil di runtime; tidak ada middleware.ts). Endpoint auth/bridge menerima POST kredensial tanpa batas percobaan dan, bila signin gagal, otomatis melakukan signUp (signup terbuka). auth/verify melakukan signInWithPassword untuk tiap request tanpa batas.
- 🎯 **Kenapa penting:** Endpoint login tanpa rate limit memungkinkan credential stuffing / brute-force password (OWASP A07 Identification and Authentication Failures). Signup terbuka tanpa batas memungkinkan pembuatan akun massal/spam dan penyalahgunaan kuota email Supabase. Meskipun Supabase Auth punya rate limit internal, endpoint ini menambah lapisan yang bisa di-hammer dan memperbesar permukaan abuse (mis. enumerasi via perilaku signin-vs-signup di bridge).
- 💥 **Dampak bila dibiarkan:** Brute-force/credential-stuffing terhadap akun pengguna, pembuatan akun spam massal, kemungkinan exhaustion kuota auth/email Supabase, dan biaya/serangan DoS pada fungsi serverless.
- 🛠 **Cara perbaiki:**
    1. Tambahkan rate limiting per-IP (dan per-email) pada auth/bridge dan auth/verify, mis. via Upstash Ratelimit atau penyimpanan token-bucket di Supabase.
    2. Pisahkan jalur signin dan signup; jangan auto-signup dalam endpoint signin tanpa kontrol (pertimbangkan verifikasi email / kontrol pendaftaran).
    3. Tambahkan backoff/penguncian sementara setelah sejumlah kegagalan untuk kombinasi IP+email.
- ✅ **Hasil cek-ulang:** NYATA. Grep seluruh app/ untuk rate-limit/throttle/upstash/ratelimit/429 = nihil (satu-satunya hit hanya passthrough string error Supabase di verify/route.ts:83). Tidak ada middleware.ts di repo (find: tidak ada). bridge/route.ts:59-88 melakukan signInWithPassword lalu fallback signUp tanpa batas percobaan; verify/route.ts:70-73 signInWithPassword tiap request tanpa batas. Memungkinkan credential stuffing/brute-force + signup massal (OWASP A07). Severity high wajar. Risk diturunkan ke medium: Supabase Auth punya rate limit internal (diakui auditor), jadi brute-force tidak sepenuhnya unbounded; endpoint ini menambah permukaan abuse tapi bukan satu-satunya garis pertahanan.

#### [#3] Endpoint translate-subtitle (DeepL berbayar) tanpa autentikasi maupun rate limit
- 🚦 **GENTING** · ⏱ ±2hr · ⚠ Risiko fix bikin rusak: **SEDANG** · 📍 `app/api/translate-subtitle/route.ts` (24-95)
- 🔍 **Masalah:** POST /api/translate-subtitle (Edge runtime) memanggil DeepL API menggunakan DEEPL_API_KEY server tanpa pengecekan auth sama sekali dan tanpa rate limit. Batas yang ada hanya per-request (MAX_CUES 2000, MAX_CHARS_TOTAL 100_000), namun tidak ada batas jumlah request per klien.
- 🎯 **Kenapa penting:** Endpoint publik tanpa auth yang memproksikan API berbayar memungkinkan penyalahgunaan kuota/biaya: penyerang dapat mengirim banyak request hingga MAX_CHARS_TOTAL masing-masing untuk menghabiskan kuota/menimbulkan biaya DeepL (OWASP A04 Insecure Design / unrestricted resource consumption). Ini juga vektor DoS biaya pada fungsi Edge.
- 💥 **Dampak bila dibiarkan:** Penghabisan kuota dan/atau tagihan DeepL yang membengkak, gangguan layanan terjemahan untuk pengguna sah, potensi biaya finansial tak terkendali.
- 🛠 **Cara perbaiki:**
    1. Wajibkan sesi terautentikasi (cookie Supabase via createClient) sebelum memanggil DeepL, atau batasi ke origin/route internal yang terkontrol.
    2. Tambahkan rate limiting per-IP/per-user dan kuota harian agregat.
    3. Pertimbangkan caching hasil terjemahan untuk cue identik guna mengurangi panggilan berulang.
- ✅ **Hasil cek-ulang:** NYATA. translate-subtitle/route.ts (Edge) hanya cek env DEEPL_API_KEY (baris 25-26) lalu memproksikan DeepL dengan Authorization DeepL-Auth-Key (63-70). Tidak ada import auth/createClient sama sekali (head file bersih dari import auth, terkonfirmasi). Satu-satunya batas: MAX_CUES 2000 + MAX_CHARS_TOTAL 100000 PER REQUEST (14-15,41,45) — tidak ada batas jumlah request per klien. DEEPL_API_KEY terkonfigurasi nyata di .env.local. Endpoint publik tanpa auth yang memproksikan API berbayar = abuse kuota/biaya + cost-DoS pada fungsi Edge (OWASP A04 unrestricted resource consumption). Severity high wajar. Risk medium: dampak finansial nyata tapi besarannya bergantung kuota/plan DeepL dan tidak ada bukti eksploitasi; bukan kompromi data.

#### [#4] Ketidakcocokan klasifikasi per-user antara route kv/sync dan RLS DB → bypass auth-gate & risiko data per-user tersimpan di ruang platform (anon-readable)
- 🚦 **PENTING** · ⏱ ±2hr · ⚠ Risiko fix bikin rusak: **SEDANG** · 📍 `app/api/kv/sync/route.ts, supabase/migrations/0008_kv_per_user_schema.sql` (kv/sync:42-59,95-107,135-171; 0008:34-50)
- 🔍 **Masalah:** Route mendefinisikan isPerUserKey hanya untuk 5 prefix (playly-prefs-/welcomed-/welcome-/onboarding-/notif-) plus playly-account-* berisi '@'. Sementara DB kv_is_per_user_key mencakup 10 pola (termasuk playly-state-*, playly-2fa-*, playly-cloud-last-sync*, playly-cloud-retry*). Untuk key yang TIDAK diklasifikasi route sebagai per-user, POST menstempel user_id:null (baris 106) sehingga data masuk ruang 'platform'. Selain itu auth-gate (cek !authUser → 401) HANYA berlaku bila isPerUserKey true; GET ?key=... dan DELETE untuk key per-user yang TIDAK dikenali route akan melewati gate 401 dan langsung query, bergantung sepenuhnya pada RLS.
- 🎯 **Kenapa penting:** Karena dua daftar tidak in-lockstep (komentar di kode sendiri menyebut sinkronisasi 'MANDATORY'), key yang sebenarnya bersifat per-user (mis. playly-state-*, playly-2fa-*) bisa ditulis sebagai baris user_id=NULL = ruang platform yang dapat dibaca lintas-user/anon (OWASP A01 Broken Access Control / A04 Insecure Design). Ini adalah persis kelas bug yang menyebabkan kebocoran orphan di migration 0013. Pertahanan akhir hanya RLS; bila migration 0008/0013 belum benar-benar di-apply, ekspos langsung terjadi.
- 💥 **Dampak bila dibiarkan:** Data sensitif per-user (state, indikator 2FA) berpotensi tersimpan di ruang platform yang dapat dibaca anon/lintas-user; auth-gate aplikasi dapat dilewati untuk key yang salah klasifikasi, memindahkan seluruh beban keamanan ke RLS yang penerapannya manual.
- 🛠 **Cara perbaiki:**
    1. Jadikan route memakai satu sumber kebenaran klasifikasi yang identik dengan kv_is_per_user_key DB (samakan ke-10 pola), idealnya via konstanta bersama atau RPC.
    2. Tolak (400) penulisan key bermotif per-user sebagai platform (user_id NULL) di sisi server, jangan hanya mengandalkan RLS.
    3. Terapkan auth-gate 401 untuk SEMUA operasi kv yang menyentuh key bermotif per-user (POST/GET/DELETE), bukan hanya yang lolos isPerUserKey versi route.
    4. Tambahkan test yang memverifikasi route dan DB sepakat untuk daftar key contoh (state/2fa/cloud).
- ✅ **Hasil cek-ulang:** Fakta kode NYATA, eksploitabilitas praktis lebih rendah dari yang disiratkan. TERKONFIRMASI: route isPerUserKey hanya 5 prefix + account-@ (kv/sync:42-59) vs DB kv_is_per_user_key 10 pola termasuk playly-state-/2fa-/cloud-last-sync/cloud-retry (0008:34-50). Untuk key tak-dikenali route, POST menstempel user_id:null (kv/sync:106) = ruang platform; auth-gate 401 HANYA aktif bila isPerUserKey true (kv/sync:95,156,198), jadi GET/DELETE key per-user tak-dikenali melewati gate dan bergantung pada RLS. Komentar kode sendiri menyebut LOCKSTEP 'MANDATORY' (kv/sync:31). MITIGASI yang menurunkan severity dari high ke medium: (1) klien sah tak pernah mengirim playly-state-*/playly-2fa-* ke endpoint ini — keduanya NO_SYNC_PREFIXES di cloud-sync.js:43-50 (state ke tabel user_state, 2fa tak pernah sync). (2) Menulis baris null butuh cookie authenticated yang valid (createClient cookie-auth), bukan anon murni; insert lolos kv_per_user_insert (0008:70-76) yang TIDAK diperketat dengan kv_is_per_user_key (berbeda dari hardening anon di 0013:69-73), sehingga gap defense-in-depth nyata tapi memerlukan akun auth. (3) Ini kelas bug yang sama dengan 0013, jadi keprihatinan auditor sah. Ini insecure-design/A04 + A01 yang valid namun bukan kebocoran anon trivial — medium lebih tepat daripada high.

#### [#5] auth/bridge melakukan signup terbuka dengan validasi password sangat longgar dan tanpa pembatasan
- 🚦 **PENTING** · ⏱ ±30min · ⚠ Risiko fix bikin rusak: **SEDANG** · 📍 `app/api/auth/bridge/route.ts` (40-88)
- 🔍 **Masalah:** Bridge hanya memvalidasi bahwa email & password tidak kosong (baris 40-45), lalu jika signin gagal langsung melakukan signUp tanpa kebijakan kompleksitas/panjang password di sisi server, tanpa verifikasi/kontrol pendaftaran, dan tanpa rate limit. Pesan error signup mentah dari Supabase dikembalikan ke klien (baris 71-74), berbeda dengan auth/verify yang sengaja menyamarkan pesan.
- 🎯 **Kenapa penting:** Signup terbuka + validasi minimal memudahkan pembuatan akun dengan password lemah dan pendaftaran massal/otomatis (OWASP A07). Membocorkan pesan error signup mentah dapat membantu enumerasi/recon (mis. membedakan 'email sudah terdaftar' vs error lain), bertolak belakang dengan praktik penyamaran yang sudah diterapkan di endpoint verify.
- 💥 **Dampak bila dibiarkan:** Proliferasi akun lemah/spam, permukaan enumerasi akun, dan inkonsistensi postur keamanan antar endpoint auth.
- 🛠 **Cara perbaiki:**
    1. Terapkan kebijakan password minimum (panjang/kompleksitas) di sisi server sebelum signUp.
    2. Samakan penanganan error agar tidak membocorkan detail mentah (selaras dengan auth/verify).
    3. Tambahkan rate limiting + (opsional) verifikasi email/kontrol pendaftaran untuk membatasi signup otomatis.
